## Endpoint

```
GET /api/health
```

Perform a comprehensive health check of the Chat2Cash system, including database, Anthropic API, and Redis queue connectivity.

## Authentication

**No authentication required.** This endpoint is public for monitoring and health check integrations.

## Response

<ResponseField name="status" type="string">
  Overall system health status:
  - `"ok"` - All systems operational
  - `"error"` - One or more critical services are down
</ResponseField>

<ResponseField name="timestamp" type="string">
  ISO 8601 timestamp of when the health check was performed.
</ResponseField>

<ResponseField name="services" type="object">
  Health status of individual services.
  
  <Expandable title="services properties">
    <ResponseField name="database" type="string">
      Database connection status:
      - `"connected"` - Database is reachable
      - `"disconnected"` - Database connection failed
      - `"unknown"` - Status could not be determined
    </ResponseField>
    
    <ResponseField name="anthropic" type="string">
      Anthropic API status:
      - `"reachable"` - API is responding
      - `"api_key_invalid"` - API key authentication failed
      - `"unreachable"` - API is down or network issue
      - `"unknown"` - Status could not be determined
    </ResponseField>
    
    <ResponseField name="queue" type="string">
      Redis/BullMQ queue status:
      - `"connected"` - Redis is reachable and queue is operational
      - `"disconnected"` - Redis connection failed
      - `"unknown"` - Status could not be determined
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="latency_ms" type="number">
  Latency in milliseconds for the Anthropic API health check (only present if API is checked).
</ResponseField>

<ResponseField name="queue" type="object">
  Detailed queue statistics (only present if Redis is connected). See [Queue Health endpoint](/api/endpoints/queue-health) for details.
  
  <Expandable title="queue properties">
    <ResponseField name="extraction" type="object">
      Extraction queue metrics (waiting, active, completed, failed, delayed).
    </ResponseField>
    
    <ResponseField name="webhook" type="object">
      Webhook queue metrics (waiting, active, failed).
    </ResponseField>
  </Expandable>
</ResponseField>

## HTTP Status Codes

- **200 OK** - All systems healthy (`status: "ok"`)
- **503 Service Unavailable** - One or more systems unhealthy (`status: "error"`)

## Example Request

```bash
curl -X GET https://your-domain.com/api/health
```

## Health Check Details

<AccordionGroup>
  <Accordion title="Database Check">
    Executes `SELECT 1` query to verify:
    - Database is reachable
    - Connection pool has available connections
    - Query execution is working
    
    **Timeout**: None (relies on connection pool timeout)
  </Accordion>
  
  <Accordion title="Anthropic API Check">
    Sends HEAD request to `https://api.anthropic.com/v1/messages` with API key to verify:
    - API is reachable
    - Network connectivity is working
    - API key is valid (checks for 401 response)
    
    **Timeout**: 3 seconds
    
    **Note**: A 405 Method Not Allowed response is considered healthy (API is reachable).
  </Accordion>
  
  <Accordion title="Queue/Redis Check">
    Calls `getQueueHealth()` to verify:
    - Redis connection is active
    - BullMQ queues are accessible
    - Queue metrics can be retrieved
    
    **Timeout**: Depends on Redis client timeout (typically 5 seconds)
  </Accordion>
</AccordionGroup>

## Integration Examples

### Kubernetes Liveness Probe

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: chat2cash-api
spec:
  containers:
  - name: api
    image: chat2cash:latest
    livenessProbe:
      httpGet:
        path: /api/health
        port: 3000
      initialDelaySeconds: 30
      periodSeconds: 10
      timeoutSeconds: 5
      failureThreshold: 3
    readinessProbe:
      httpGet:
        path: /api/health
        port: 3000
      initialDelaySeconds: 10
      periodSeconds: 5
      timeoutSeconds: 3
      failureThreshold: 2
```

### Docker Health Check

```dockerfile
FROM node:20-alpine

# ... your Dockerfile commands ...

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/api/health || exit 1
```

### Uptime Monitoring (UptimeRobot, Pingdom, etc.)

Configure your monitoring service to:
1. **URL**: `https://your-domain.com/api/health`
2. **Method**: GET
3. **Expected Status Code**: 200
4. **Check Interval**: 1-5 minutes
5. **Alert on**: Status code 503 or timeout

### JavaScript Monitoring Script

```javascript
const checkHealth = async () => {
  try {
    const response = await fetch('https://your-domain.com/api/health');
    const health = await response.json();
    
    if (health.status === 'error') {
      console.error('🚨 System unhealthy:', health.services);
      
      // Alert on specific services
      if (health.services.database === 'disconnected') {
        console.error('❌ Database is down!');
        // Send PagerDuty alert
      }
      
      if (health.services.anthropic === 'unreachable') {
        console.warn('⚠️ Anthropic API is unreachable');
        // Send warning notification
      }
      
      if (health.services.queue === 'disconnected') {
        console.error('❌ Redis/Queue is down!');
        // Send critical alert
      }
    } else {
      console.log('✅ All systems operational');
      console.log(`🚀 API latency: ${health.latency_ms}ms`);
    }
    
    return health;
  } catch (error) {
    console.error('💥 Health check failed:', error.message);
    // Send critical alert - API is completely down
  }
};

// Check every 30 seconds
setInterval(checkHealth, 30000);
```

### Prometheus Metrics Export

```javascript
const { register, Gauge } = require('prom-client');

const healthGauge = new Gauge({
  name: 'chat2cash_health_status',
  help: 'System health status (1 = healthy, 0 = unhealthy)',
  labelNames: ['service']
});

const latencyGauge = new Gauge({
  name: 'chat2cash_api_latency_ms',
  help: 'Anthropic API latency in milliseconds'
});

const updateHealthMetrics = async () => {
  const response = await fetch('https://your-domain.com/api/health');
  const health = await response.json();
  
  // Set health gauges (1 = healthy, 0 = unhealthy)
  healthGauge.set(
    { service: 'database' },
    health.services.database === 'connected' ? 1 : 0
  );
  
  healthGauge.set(
    { service: 'anthropic' },
    health.services.anthropic === 'reachable' ? 1 : 0
  );
  
  healthGauge.set(
    { service: 'queue' },
    health.services.queue === 'connected' ? 1 : 0
  );
  
  if (health.latency_ms) {
    latencyGauge.set(health.latency_ms);
  }
};

// Update every 15 seconds
setInterval(updateHealthMetrics, 15000);
```

## Rate Limiting

**No rate limiting** - This endpoint is exempt from rate limiting to ensure monitoring systems can check health frequently.

## Notes

<Info>
  This endpoint is designed for automated monitoring and should not require authentication. Exposing service health status is standard practice for operational visibility.
</Info>

<Warning>
  The Anthropic API check sends a real request and counts toward your API quota. However, it uses a lightweight HEAD request to minimize impact.
</Warning>

<Tip>
  Monitor the `latency_ms` field to detect degraded Anthropic API performance before it causes user-facing issues.
</Tip>

<Note>
  Queue statistics are included when Redis is connected. Use this to spot growing backlogs or failing jobs during health checks.
</Note>
