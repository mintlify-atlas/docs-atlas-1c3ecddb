Chat2Cash automatically redacts Personally Identifiable Information (PII) from API responses based on user permissions. The system uses `google-libphonenumber` for international phone number detection and configurable regex patterns for other sensitive data types.

## Phone Number Redaction

Unlike hardcoded regex patterns, phone number detection uses `google-libphonenumber` for accurate international parsing:

```typescript
// From piiRedactor.ts:14-33
export function redactPhoneNumbers(text: string): string {
  const matches = text.match(/[+]?[\d\s\-()]{7,20}/g);
  if (!matches) return text;

  for (const match of matches) {
    for (const region of PHONE_REGIONS) {
      try {
        const parsed = phoneUtil.parse(match.trim(), region);
        if (phoneUtil.isValidNumber(parsed)) {
          text = text.replace(match, "[PHONE REDACTED]");
          break;
        }
      } catch {
        // Not a valid phone number for this region
      }
    }
  }

  return text;
}
```

**Supported regions:**
- India (IN)
- United States (US)
- United Kingdom (GB)
- Canada (CA)
- Australia (AU)
- Germany (DE)
- France (FR)
- Japan (JP)
- Singapore (SG)

## Permission-Based Access Control

Redaction is controlled by the `VIEW_PII` permission. Users with this permission see unredacted data.

```typescript
// From piiRedactor.ts:83-112
export const redactPII = async (req: Request, res: Response, next: NextFunction) => {
  const originalJson = res.json.bind(res);

  res.json = ((data: any) => {
    (async () => {
      try {
        let shouldRedact = true;

        if (req.user?.id && req.orgId) {
          shouldRedact = !(await hasPermission(
            req.user.id,
            req.orgId,
            PERMISSIONS.VIEW_PII,
          ));
        }

        if (shouldRedact) {
          data = redactSensitiveData(data);
        }

        originalJson(data);
      } catch {
        // Secure fallback: redact on any permission-check failure
        originalJson(redactSensitiveData(data));
      }
    })();
  }) as any;

  next();
};
```

**Secure by default:** If the permission check fails for any reason, the middleware defaults to redacting PII.

## Recursive Redaction

The system recursively traverses nested objects and arrays:

```typescript
// From piiRedactor.ts:53-77
export function redactSensitiveData(data: any): any {
  if (data === null || data === undefined) return data;

  if (typeof data === "string") {
    return redactString(data);
  }

  if (Array.isArray(data)) {
    return data.map((item) => redactSensitiveData(item));
  }

  if (typeof data === "object") {
    const redacted: any = {};
    for (const [key, value] of Object.entries(data)) {
      if (SENSITIVE_KEYS.has(key) && typeof value === "string") {
        redacted[key] = "[REDACTED]";
      } else {
        redacted[key] = redactSensitiveData(value);
      }
    }
    return redacted;
  }

  return data;
}
```

## Usage in Routes

```typescript
import { redactPII } from "./middlewares/piiRedactor";

// Apply PII redaction to all routes
app.use("/api", redactPII, router);

// Or apply selectively
router.get("/orders/:id", authHandler, requireOrg, redactPII, getOrderHandler);
```

## Logging Redaction

PII is also redacted from logs using Pino's built-in `redact` feature:

```typescript
// From logger.ts:28-31
redact: {
  paths: SENSITIVE_KEYS.flatMap(key => [`req.body.${key}`, `res.body.${key}`, `*.${key}`]),
  censor: "***REDACTED***",
}
```

This ensures sensitive data never appears in log files or observability platforms.

## Best Practices

<Steps>
  <Step title="Always apply redactPII middleware">
    Add `redactPII` to all routes that return user-generated content or customer data.
  </Step>
  
  <Step title="Use permission-based access">
    Grant `VIEW_PII` only to roles that require it (e.g., admins, support agents with audit trails).
  </Step>
  
  <Step title="Test with real data">
    Validate redaction patterns against production-like data to catch edge cases.
  </Step>
  
  <Step title="Log redaction events">
    Consider logging when a user with `VIEW_PII` accesses sensitive data for compliance audits.
  </Step>
  
  <Step title="Extend patterns as needed">
    Add new patterns to `PII_PATTERNS` for jurisdiction-specific ID types (e.g., Brazilian CPF, Canadian SIN).
  </Step>
</Steps>

## Extending Patterns

To add a new PII pattern, edit `src/config/piiPatterns.ts`:

```typescript
export const PII_PATTERNS: PIIPattern[] = [
  // ... existing patterns
  {
    name: "brazilian_cpf",
    regex: /\b\d{3}\.\d{3}\.\d{3}-\d{2}\b/g,
    replacement: "[CPF REDACTED]",
  },
];
```

No code changes required in the middleware — patterns are automatically applied.
