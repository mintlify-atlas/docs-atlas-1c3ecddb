title: Rate Limiting
description: Tier-based rate limiting with Redis caching and dynamic limit adjustment

Chat2Cash implements organization-tier-based rate limiting using `express-rate-limit` with Redis-backed tier caching to prevent N+1 database queries.

## Rate Limiter Types

### extractLimiter

Applies strict rate limits for computationally expensive operations like AI extraction.

```typescript
// Usage in routes
router.post("/orders/extract",
  authHandler,
  requireOrg,
  extractLimiter,
  extractOrderHandler
);
```

**Limits per tier:**
- **Free:** 10 requests per 15 minutes (configurable via `RATE_LIMIT_FREE`)
- **Pro:** 100 requests per 15 minutes (configurable via `RATE_LIMIT_PRO`)
- **Enterprise:** 1000 requests per 15 minutes (configurable via `RATE_LIMIT_ENTERPRISE`)

### generalLimiter

Applies relaxed rate limits for read-heavy operations with a 5x multiplier.

```typescript
// Usage in routes
router.get("/orders",
  authHandler,
  requireOrg,
  generalLimiter,
  getOrdersHandler
);
```

**Limits per tier:**
- **Free:** 50 requests per 15 minutes (5x base)
- **Pro:** 500 requests per 15 minutes (5x base)
- **Enterprise:** 5000 requests per 15 minutes (5x base)

## Implementation

### Tier-Based Limit Resolution

The system fetches organization tiers from the database and caches them in Redis for 5 minutes to avoid repeated lookups:

```typescript
// From rateLimiter.ts:26-60
async function getOrgRateLimit(orgId: string | undefined): Promise<number> {
  if (!orgId) return TIER_LIMITS.free;

  const cacheKey = `org:tier:${orgId}`;

  // 1. Try Redis cache first
  try {
    const cached = await redis.get(cacheKey);
    if (cached && TIER_LIMITS[cached] !== undefined) {
      return TIER_LIMITS[cached];
    }
  } catch {
    // Redis unavailable — fall through to DB
  }

  // 2. Cache miss: query DB then populate cache
  try {
    const org = await db
      .select({ tier: organizationsTable.tier })
      .from(organizationsTable)
      .where(eq(organizationsTable.id, orgId))
      .limit(1);

    if (org.length > 0 && org[0].tier) {
      const tier = org[0].tier;
      // Populate Redis cache; ignore errors so a Redis blip doesn't break auth
      redis.setex(cacheKey, TIER_CACHE_TTL, tier).catch(() => {});
      return TIER_LIMITS[tier] ?? TIER_LIMITS.free;
    }
  } catch {
    // Fall back to free tier on DB lookup failure
  }

  return TIER_LIMITS.free;
}
```

### Dynamic Limiter Instance Pooling

Instead of creating a new rate limiter instance per request, the system pools limiters by their `max` value:

```typescript
// From rateLimiter.ts:62-80
const limiterCache = new Map<number, ReturnType<typeof rateLimit>>();

function getLimiter(max: number) {
  if (limiterCache.has(max)) return limiterCache.get(max)!;

  const limiter = rateLimit({
    windowMs,
    max,
    standardHeaders: true,  // Return RateLimit-* headers
    legacyHeaders: false,   // Disable X-RateLimit-* headers
    keyGenerator: (req: Request) => req.orgId ?? req.ip ?? "unknown",
    message: {
      error: `Rate limit exceeded. Your plan allows ${max} requests per ${windowMs / 60000} minutes.`,
    },
  });

  limiterCache.set(max, limiter);
  return limiter;
}
```

### Middleware Implementation

```typescript
// From rateLimiter.ts:85-96
export const extractLimiter = async (req: Request, res: Response, next: NextFunction) => {
  const limit = await getOrgRateLimit(req.orgId);
  getLimiter(limit)(req, res, next);
};

export const generalLimiter = async (req: Request, res: Response, next: NextFunction) => {
  const baseLimit = await getOrgRateLimit(req.orgId);
  getLimiter(baseLimit * 5)(req, res, next);
};
```

## Configuration

Rate limits are configured via environment variables:

```bash
# Window duration (milliseconds)
RATE_LIMIT_WINDOW_MS=900000  # 15 minutes

# Per-tier limits (requests per window)
RATE_LIMIT_FREE=10
RATE_LIMIT_PRO=100
RATE_LIMIT_ENTERPRISE=1000
```

## Response Headers

When rate limiting is active, the API returns standard `RateLimit-*` headers:

```http
HTTP/1.1 200 OK
RateLimit-Limit: 100
RateLimit-Remaining: 87
RateLimit-Reset: 1709582400
```

**When the limit is exceeded:**

```http
HTTP/1.1 429 Too Many Requests
RateLimit-Limit: 100
RateLimit-Remaining: 0
RateLimit-Reset: 1709582400
Retry-After: 600

{
  "error": "Rate limit exceeded. Your plan allows 100 requests per 15 minutes."
}
```

## Architecture Decisions

<AccordionGroup>
  <Accordion title="Why cache tier in Redis?">
    Without caching, every authenticated request would trigger a database lookup to fetch the organization's tier (N+1 problem at the middleware level). Redis caching reduces latency and database load by storing tier information for 5 minutes.
  </Accordion>

  <Accordion title="Why pool limiter instances?">
    Creating a new `rateLimit()` instance per request is expensive. Pooling by `max` value reuses instances across organizations in the same tier.
  </Accordion>

  <Accordion title="What happens if Redis is down?">
    The middleware falls back to the database. If the database is also unavailable, it defaults to the free tier to prevent service disruption.
  </Accordion>

  <Accordion title="Why use orgId instead of IP?">
    Organization-based rate limiting provides better isolation for multi-tenant applications. An organization can't exhaust its quota by making requests from multiple IPs.
  </Accordion>
</AccordionGroup>

## Key Generator Logic

The rate limiter uses the following precedence for key generation:

```typescript
keyGenerator: (req: Request) => req.orgId ?? req.ip ?? "unknown"
```

1. **`req.orgId`** (set by `authHandler` for authenticated requests)
2. **`req.ip`** (fallback for unauthenticated requests)
3. **`"unknown"`** (fallback if both are missing)

## Monitoring Rate Limits

You can track rate limit consumption by parsing response headers:

```typescript
const response = await fetch("https://api.chat2cash.dev/api/orders/extract", {
  method: "POST",
  headers: { "x-api-key": "c2c_live_abc123" },
  body: JSON.stringify({ message: "..." }),
});

const limit = response.headers.get("RateLimit-Limit");
const remaining = response.headers.get("RateLimit-Remaining");
const reset = response.headers.get("RateLimit-Reset");

console.log(`Used ${limit - remaining}/${limit} requests. Resets at ${new Date(reset * 1000)}`);
```

## Common Patterns

### Apply Different Limits to Different Routes

```typescript
// Strict limit for AI extraction
router.post("/orders/extract", authHandler, requireOrg, extractLimiter, handler);

// Relaxed limit for read operations
router.get("/orders", authHandler, requireOrg, generalLimiter, handler);
router.get("/orders/:id", authHandler, requireOrg, generalLimiter, handler);

// No rate limiting for health checks
router.get("/health", healthCheckHandler);
```

### Bypass Rate Limiting for Specific Organizations

Modify `getOrgRateLimit` to return `Infinity` for whitelisted organizations:

```typescript
const UNLIMITED_ORGS = new Set(["org_partner_123", "org_internal"]);

async function getOrgRateLimit(orgId: string | undefined): Promise<number> {
  if (!orgId) return TIER_LIMITS.free;
  if (UNLIMITED_ORGS.has(orgId)) return Infinity;
  
  // ... existing logic
}
```
