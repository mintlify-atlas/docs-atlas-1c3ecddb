## Multi-Tenancy Overview

Chat2Cash is a **multi-tenant SaaS application** where each organization has its own isolated data. All database queries enforce tenant isolation via `organization_id` foreign keys.

**Isolation Model:** Shared database, shared schema, row-level filtering  
**Root Entity:** `organizations` table  
**Enforcement:** Application-level (Drizzle ORM queries)

## Tenant Hierarchy

```
organizations (tenant root)
    ├─► business_profiles (1:1)
    ├─► customers (1:many)
    ├─► products (1:many)
    ├─► orders (1:many)
    │      └─► order_items (1:many)
    ├─► users (1:many)
    ├─► api_keys (1:many)
    └─► roles (1:many)
```

Every data record belongs to exactly one organization. Queries filter by `organization_id` to prevent cross-tenant data leaks.

## Authentication Methods

Chat2Cash supports two authentication methods:

1. **JWT (Neon Auth)** - Human users via OAuth/email
2. **API Keys** - Machine-to-machine access

Both methods set `req.orgId` on the Express request object, enabling tenant filtering.

---

## JWT Authentication (Neon Auth)

### Flow

```
1. User logs in via Neon Auth (OAuth/email)
2. Neon Auth issues JWT with claims: { sub: userId, email, name }
3. Client sends: Authorization: Bearer <jwt>
4. API verifies JWT signature using Neon JWKS endpoint
5. API upserts user to local `users` table (JIT provisioning)
6. API sets: req.user = user, req.orgId = user.organizationId
7. API enforces: WHERE organizationId = req.orgId
```

**Code:** `backend/src/middlewares/authHandler.ts:60-94`

### JWT Verification

```typescript
import { createRemoteJWKSet, jwtVerify } from "jose";

const JWKS = createRemoteJWKSet(new URL(env.NEON_JWKS_URL));

export const authHandler = async (req: Request, res: Response, next: NextFunction) => {
  const authHeader = req.headers.authorization;

  if (authHeader?.startsWith("Bearer ")) {
    const token = authHeader.split(" ")[1];

    try {
      const { payload } = await jwtVerify(token, JWKS);

      const userId = payload.sub as string;
      const userEmail = payload.email as string;

      // Atomic JIT sync: upsert user in local DB on first login
      const [user] = await db
        .insert(usersTable)
        .values({
          id: userId,
          email: userEmail,
          name: (payload.name as string) ?? "Unknown",
        })
        .onConflictDoUpdate({
          target: usersTable.id,
          set: {
            email: userEmail,
            name: (payload.name as string) ?? "Unknown",
          },
        })
        .returning();

      req.user = user;

      // Set org context from the user record
      if (req.user?.organizationId) {
        req.orgId = req.user.organizationId;
      }

      return next();
    } catch (jwtError) {
      throw new AppError("Invalid or Expired Token", 401);
    }
  }

  throw new AppError("Authentication required", 401);
};
```

**Code:** `backend/src/middlewares/authHandler.ts:60-100`

**Key Points:**

1. **Remote JWKS:** API fetches Neon's public keys to verify JWT signatures
2. **JIT Provisioning:** Users are upserted on first login (no pre-registration)
3. **Atomic Upsert:** `INSERT ... ON CONFLICT DO UPDATE` prevents race conditions
4. **Org Assignment:** Admin must assign `organizationId` via dashboard (NULL by default)

## API Key Authentication

### Flow

```
1. Admin generates API key via dashboard
2. API computes SHA-256 hash, stores in `api_keys` table
3. API returns plaintext key to admin (shown once)
4. Client sends: X-API-Key: sk-abc123...
5. API computes SHA-256(sk-abc123...)
6. API queries: SELECT * FROM api_keys WHERE key_hash = <hash> AND is_active = true
7. API verifies hash using constant-time comparison
8. API sets: req.orgId = apiKey.organizationId
9. API enforces: WHERE organizationId = req.orgId
```

### Security: Constant-Time Comparison

**Why Constant-Time?**

Prevents **timing attacks** where an attacker infers hash byte values by measuring response times.

```typescript
import crypto from "crypto";

// Compute hash as a Buffer so we can use timingSafeEqual below
const incomingHash = crypto.createHash("sha256").update(apiKeyHeader).digest();
const incomingHashHex = incomingHash.toString("hex");

// Fetch by hash (indexed column)
const result = await db
  .select({ orgId: apiKeysTable.organizationId, keyHash: apiKeysTable.keyHash })
  .from(apiKeysTable)
  .where(and(eq(apiKeysTable.keyHash, incomingHashHex), eq(apiKeysTable.isActive, true)))
  .limit(1);

if (result.length === 0) throw new AppError("Invalid API Key", 401);

// Constant-time comparison — prevents timing side-channel attacks
const storedHash = Buffer.from(result[0].keyHash, "hex");
if (
  storedHash.length !== incomingHash.length ||
  !crypto.timingSafeEqual(incomingHash, storedHash)
) {
  throw new AppError("Invalid API Key", 401);
}

req.orgId = result[0].orgId;
return next();
```

**Code:** `backend/src/middlewares/authHandler.ts:29-56`

**Attack Scenario Without Constant-Time:**

```
Attacker tries: sk-aaa... → 50ms response (hash mismatch detected early)
Attacker tries: sk-abc... → 52ms response (hash match for first byte, more checks)
→ Attacker infers first byte is 'a', then 'b', then 'c'
```

**With Constant-Time:**

```
All invalid keys → 50ms response (same time for all mismatches)
→ Attacker cannot infer hash byte values
```

### API Key Generation (Example)

```typescript
import crypto from "crypto";

function generateApiKey(): string {
  const randomBytes = crypto.randomBytes(32);
  return `sk-${randomBytes.toString("base64url")}`;
}

function hashApiKey(key: string): string {
  return crypto.createHash("sha256").update(key).digest("hex");
}

// Generate key
const apiKey = generateApiKey(); // sk-abc123...
const keyHash = hashApiKey(apiKey); // SHA-256 hash

// Store in database
await db.insert(apiKeysTable).values({
  id: "key_xyz",
  organizationId: "org_abc",
  keyHash,
  name: "Production API Key",
  maskedKey: `sk-...${apiKey.slice(-6)}`, // sk-...abc123
  isActive: true,
});

// Return plaintext key to user (shown once)
return { apiKey };
```

**Never Stored:** The plaintext key is never stored in the database. Only the hash is stored.

## Tenant Isolation Enforcement

### Application-Level Filtering

**Every query includes `organization_id` in WHERE clause:**

```typescript
// Example: Fetch orders for current org
const orders = await db
  .select()
  .from(ordersTable)
  .where(eq(ordersTable.organizationId, req.orgId)); // ← Tenant filter
```

**Why Application-Level?**

- Simpler than PostgreSQL Row-Level Security (RLS)
- Works with Drizzle ORM's type-safe query builder
- Easier to debug and audit

**Future:** Add PostgreSQL RLS policies as defense-in-depth.

---

### Example: Fetch Orders (Controller)

```typescript
import { db } from "../config/db";
import { ordersTable, orderItemsTable } from "../schema";
import { eq, and, isNull } from "drizzle-orm";

export const getOrders = async (req: Request, res: Response) => {

    .select()
    .from(ordersTable)

      and(
        eq(ordersTable.organizationId, orgId!), // ← Tenant filter
```

### Example: Create Order (Controller)

```typescript
export const extractOrder = async (req: Request, res: Response) => {
  const { orgId } = req; // Set by authHandler middleware
  const { message } = req.body;

  // Extract order via Anthropic API
  const order = await anthropicService.extractOrderFromMessage(message);

  // Save order to database
  const savedOrder = await db.transaction(async (tx) => {
    // Create or find customer
    let customer;
    if (order.customerPhone) {
      [customer] = await tx
        .select()
        .from(customersTable)
        .where(
          and(
            eq(customersTable.organizationId, orgId!), // ← Tenant filter
            eq(customersTable.phone, order.customerPhone)
          )
        )
        .limit(1);

      if (!customer) {
        [customer] = await tx
          .insert(customersTable)
          .values({
            id: generateId("cust"),
            organizationId: orgId!, // ← Tenant isolation
            name: order.customerName,
            phone: order.customerPhone,
          })
          .returning();
      }
    }

    // Insert order
    const [newOrder] = await tx
      .insert(ordersTable)
      .values({
        id: generateId("ord"),
        organizationId: orgId!, // ← Tenant isolation
        customerId: customer.id,
        // ...
      })
      .returning();

    return newOrder;
  });

  res.json({ order: savedOrder });
};
```

**Key Points:**

1. `organizationId` is set from `req.orgId` for all inserts
2. Customer lookup includes `eq(customersTable.organizationId, orgId)`
3. Transaction ensures atomicity (order + customer created together)

```typescript
export const requirePermission = (permission: string) => {
  return async (req: Request, res: Response, next: NextFunction) => {
    const { user, orgId } = req;

    if (!user || !orgId) {
      return next(new AppError("Authentication required", 401));
    }

    // Fetch role permissions
    const [role] = await db
      .select()
      .from(rolesTable)
      .where(
        and(
          eq(rolesTable.organizationId, orgId),
          eq(rolesTable.name, user.role)
        )
      )
      .limit(1);

    if (!role || !role.permissions.includes(permission)) {
      return next(new AppError("Insufficient permissions", 403));
    }

    next();
  };
};
```

## Cross-Tenant Attack Scenarios

### Scenario 1: Direct Database Query (Prevented)

**Attack:**

```typescript
// Attacker tries to access another org's orders
const orders = await db
  .select()
  .from(ordersTable)
  .where(eq(ordersTable.id, "ord_from_another_org"));
```

**Prevented By:**

Query must include `eq(ordersTable.organizationId, req.orgId)` to pass code review.

**Defense-in-Depth (Future):** PostgreSQL RLS policies enforce `organization_id` at database level.

---

### Scenario 2: Job Queue Poisoning (Prevented)

**Attack:**

```typescript
// Attacker enqueues job with another org's orgId
await addExtractionJob({
  type: "chat_log",
  orgId: "org_victim", // ← Attacker controls this
  messages: [...],
});
```

**Prevented By:**

1. API sets `orgId: req.orgId` (from authenticated JWT/API key)
2. Attacker cannot forge JWT (signed by Neon Auth)
3. Attacker cannot forge API key (SHA-256 hash verified)

**Code:** `backend/src/services/queueService.ts:275-284`

---

### Scenario 3: API Key Reuse (Prevented)

**Attack:**

```typescript
// Attacker steals API key from Org A, uses it to access Org B's data
fetch("/orders?orgId=org_b", {
  headers: { "X-API-Key": "sk-stolen-from-org-a" },
});
```

**Prevented By:**

1. API key is hashed and stored with `organizationId = org_a`
2. Auth middleware sets `req.orgId = org_a` (from API key lookup)
3. Query parameter `orgId=org_b` is ignored (only `req.orgId` is used)

**Code:** `backend/src/middlewares/authHandler.ts:55`

## Security Best Practices

### 1. Never Trust Client Input

**Bad:**

```typescript
const orgId = req.body.orgId; // ← Attacker-controlled
```

**Good:**

```typescript
const orgId = req.orgId; // ← Set by authHandler middleware
```

---

### 2. Always Filter by `organizationId`

**Bad:**

```typescript
const order = await db
  .select()
  .from(ordersTable)
  .where(eq(ordersTable.id, orderId)) // ← Cross-tenant access possible
  .limit(1);
```

**Good:**

```typescript
const order = await db
  .select()
  .from(ordersTable)
  .where(
    and(
      eq(ordersTable.id, orderId),
      eq(ordersTable.organizationId, req.orgId) // ← Tenant filter
    )
  )
  .limit(1);
```

---

### 3. Use Constant-Time Comparison for Secrets

**Bad:**

```typescript
if (incomingHash === storedHash) { // ← Timing attack possible
  // ...
}
```

```typescript
  // ...
}
```

## Compliance & Privacy

### GDPR Compliance

**Right to Erasure ("Right to be Forgotten"):**

1. User requests account deletion
2. Admin soft-deletes user: `UPDATE users SET deleted_at = NOW()`
3. Cron job hard-deletes after 30 days: `DELETE FROM users WHERE deleted_at < NOW() - INTERVAL '30 days'`

**Data Export:**

```typescript

  const { orgId } = req;

  const [orders, customers, products] = await Promise.all([
    db.select().from(ordersTable).where(eq(ordersTable.organizationId, orgId)),
    db.select().from(customersTable).where(eq(customersTable.organizationId, orgId)),
    db.select().from(productsTable).where(eq(productsTable.organizationId, orgId)),
  ]);

  res.json({ orders, customers, products });
});
```

## Troubleshooting

### User Cannot Access Any Data

**Cause:** `user.organizationId = NULL`

**Fix:** Admin assigns user to org via dashboard.

---

### Cross-Tenant Data Leak

**Cause:** Query missing `eq(table.organizationId, req.orgId)` filter.

**Fix:** Add tenant filter to all queries. Enable PostgreSQL RLS for defense-in-depth.

---

### API Key Rejected (401)

**Cause:** Key revoked (`is_active = false`) or invalid hash.

**Fix:**

1. Check key status: `SELECT * FROM api_keys WHERE key_hash = <hash>`
2. Regenerate key if revoked
