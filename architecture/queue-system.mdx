## Queue Architecture

Chat2Cash uses **BullMQ** (Redis-backed job queue) to decouple API request handling from compute-intensive AI processing. This architecture enables:

- **Async processing:** API returns immediately with a job ID
- **Horizontal scaling:** Add more workers without changing the API
- **Fault tolerance:** Jobs persist in Redis and retry on failure
- **Rate limiting:** Respect Anthropic API tier limits

**Queue Library:** BullMQ (v5)  
**Backend:** Redis 7 with AOF persistence  
**Implementation:** `backend/src/services/queueService.ts`

## Queue Overview

### Queue Definitions

```typescript
export const extractionQueue = new Queue("order-extraction", {
  connection,

    attempts: 3,
    backoff: { type: "exponential", delay: 3000 },

    removeOnFail: false,
  },

export const webhookQueue = new Queue("webhook-delivery", {
  connection,
  defaultJobOptions: {
    attempts: 10,
    backoff: { type: "exponential", delay: 5000 },
    removeOnComplete: { age: 3600 * 24 },
    removeOnFail: { age: 3600 * 72 },
  },
});
```

**Code:** `backend/src/services/queueService.ts:29-47`

| Queue | Purpose | Max Retries | Backoff | Retention |
|-------|---------|-------------|---------|------------|
| `order-extraction` | AI order extraction | 3 attempts | 3s → 9s → 27s (exponential) | Completed: 24h, Failed: Forever (DLQ) |
| `webhook-delivery` | Async webhook delivery | 10 attempts | 5s → 15s → 45s → ... (exponential) | Completed: 24h, Failed: 72h |

**Why Separate Queues?**

- **Different retry policies:** Extraction jobs fail fast (AI errors unlikely to resolve), webhooks retry aggressively (transient network errors)
- **Priority handling:** Extraction queue has rate limiting; webhooks don't
- **Observability:** Separate metrics and DLQ for each queue

## Redis Configuration

### AOF Persistence

**Docker Compose Configuration:**

```yaml
redis:
  image: redis:7-alpine
  command: redis-server --appendonly yes --appendfsync everysec
  volumes:
    - redis_data:/data
```

**Code:** `docker-compose.yml:22-31`

**AOF Settings:**

| Setting | Value | Purpose |
|---------|-------|----------|
| `--appendonly yes` | Enabled | Write all commands to append-only file |
| `--appendfsync everysec` | Every second | Fsync to disk every 1 second (balance between durability and performance) |

**Durability Guarantee:**

- **Crash at T+0.5s:** Last 0.5s of jobs may be lost
- **Crash at T+1s:** All jobs up to T+0s are persisted
- **Restart:** Redis replays AOF file to restore queue state

**Alternative: `--appendfsync always`** (slower but zero data loss)

### Job Processing Flow

```typescript
async (job: Job<ExtractionJobData, ExtractionJobResult>) => {
  const cid = job.data.correlationId ?? job.id;
  logger.info({ jobId: job.id, type: job.data.type, correlationId: cid }, "Processing extraction job");

  await job.updateProgress(10);

  let savedOrder;
  const { orgId } = job.data;
  if (!orgId) throw new Error("Invalid job data: missing orgId");

  if (job.data.type === "single_message" && job.data.message) {
    const order = await anthropicService.extractOrderFromMessage(job.data.message);
    await job.updateProgress(70);
    savedOrder = await storage.addOrder(orgId, order);
  } else if (job.data.type === "chat_log" && job.data.messages) {
    const order = await anthropicService.extractOrderFromChat(job.data.messages);
    await job.updateProgress(70);
    savedOrder = await storage.addChatOrder(orgId, order);
  } else {
    throw new Error("Invalid job data: missing message or messages");
  }

  await job.updateProgress(90);

  // Enqueue webhook delivery to a separate queue
  if (job.data.webhookUrl) {
    await webhookQueue.add("deliver", {
      webhookUrl: job.data.webhookUrl,
      correlationId: cid,
      payload: {
        jobId: job.id,
        status: "completed",
        orderId: savedOrder.id,
        order: savedOrder,
      },
    });
    logger.info({ jobId: job.id, correlationId: cid }, "Webhook delivery enqueued");
  }

  await job.updateProgress(100);

  return { orderId: savedOrder.id, status: "completed" };
}
```

**Code:** `backend/src/services/queueService.ts:79-122`

**Job Lifecycle:**

1. **Start (Progress: 10%)** - Job picked up by worker
2. **AI Extraction (Progress: 70%)** - Anthropic API call completes
3. **Database Save (Progress: 90%)** - Order saved to PostgreSQL
4. **Webhook Enqueue (Progress: 100%)** - Webhook job added to `webhook-delivery` queue (if provided)
5. **Complete** - Job result returned

**Error Handling:**

- `throw new Error()` → Job marked as "failed" and retried
- After 3 attempts → Job moves to DLQ (dead letter queue)

```typescript
worker.on("failed", (job, err) => {
  const isFinalAttempt = job !== undefined &&
    job.attemptsMade >= (job.opts.attempts ?? 1);

  if (isFinalAttempt) {
    // Alert on hard-fail (all retries exhausted) so the team sees it immediately
    logger.error({ jobId: job?.id, err }, "Extraction job hard-failed (DLQ)");
    if (env.SENTRY_DSN) {
      Sentry.captureException(err, {
        tags: { queue: "order-extraction", jobId: job?.id ?? "unknown" },
        extra: {
          orgId: job?.data.orgId,
          type: job?.data.type,
          correlationId: job?.data.correlationId,
          attemptsMade: job?.attemptsMade,
        },
      });
    }
  } else {
    logger.warn({ jobId: job?.id, attempt: job?.attemptsMade, err }, "Extraction job failed (will retry)");
  }

  // Enqueue failure webhook so the worker doesn't block
  if (job?.data.webhookUrl) {
    webhookQueue.add("deliver", {
      webhookUrl: job.data.webhookUrl,
      correlationId: job.data.correlationId ?? job.id,
      payload: {
        jobId: job.id,
        status: "failed",
        error: err.message,
      },
    }).catch((enqueueErr) => {
      logger.error({ jobId: job.id, err: enqueueErr }, "Failed to enqueue failure webhook");
    });
  }
});
```

### Webhook Delivery Logic

```typescript
async (job: Job<WebhookJobData>) => {
  const { webhookUrl, payload, correlationId } = job.data;
  logger.info({ jobId: job.id, webhookUrl, correlationId }, "Delivering webhook");

  const response = await fetch(webhookUrl, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "X-Correlation-Id": correlationId ?? "",
    },
    body: JSON.stringify(payload),
    signal: AbortSignal.timeout(10_000), // 10s timeout
  });

  if (!response.ok) {
    const status = response.status;

    // 4xx errors (except 408 Request Timeout and 429 Too Many Requests) are
    // permanent client-side failures — retrying will never succeed, so drop
    // the job immediately to avoid wasting up to 10 retry attempts.
    const isRetryable =
      status === 408 ||   // Request Timeout
      status === 429 ||   // Too Many Requests
      status >= 500;      // Server Error (transient)

    if (!isRetryable) {
      throw new UnrecoverableError(
        `Webhook permanently failed with ${status} ${response.statusText} — dropping job (non-retryable client error)`
      );
    }

    throw new Error(`Webhook returned ${status}: ${response.statusText}`);
  }

  logger.info({ jobId: job.id, correlationId }, "Webhook delivered successfully");
}
```

**Code:** `backend/src/services/queueService.ts:191-226`

**Retry Logic:**

| HTTP Status | Retry? | Reason |
|-------------|--------|--------|
| `200-299` | No | Success |
| `400-407, 410-428, 430-499` | No | Permanent client error (bad request) |
| `408` (Request Timeout) | Yes | Transient network issue |
| `429` (Too Many Requests) | Yes | Rate limit (backoff will help) |
| `500-599` | Yes | Transient server error |

**Why Drop Non-Retryable Errors?**

- Avoids wasting 10 retry attempts on a 404 or 401
- Uses BullMQ's `UnrecoverableError` to immediately move job to "failed" state

**Code:** `backend/src/services/queueService.ts:206-222`

## Job Lifecycle

### Job States

```
waiting → active → completed
                 ↘ failed → waiting (retry)
                          ↘ failed (DLQ after max attempts)
```

| State | Description |
|-------|-------------|
| `waiting` | Job enqueued, waiting for worker |
| `active` | Worker processing job |
| `completed` | Job succeeded, result returned |
| `failed` | Job failed, will retry or move to DLQ |
| `delayed` | Job scheduled for future execution (not used in Chat2Cash) |

## Performance Tuning

### Concurrency vs. Rate Limits

**Current Settings:**

- **Extraction worker concurrency:** 3 jobs per worker
- **Rate limit:** 10 jobs/minute (Anthropic tier 1 limit)

**Scaling Example:**

| Worker Replicas | Total Concurrency | Jobs/Minute | Avg Job Time |
|-----------------|-------------------|-------------|---------------|
| 1 | 3 | 10 | ~18s |
| 2 | 6 | 10 | ~9s |
| 5 | 15 | 10 | ~4s |

**Bottleneck:** Rate limiter (10 jobs/min) becomes the bottleneck beyond 2 replicas.

**To Scale Further:**

1. Upgrade Anthropic tier (higher rate limits)
2. Increase `limiter.max` in `queueService.ts:127`

---

### Redis Memory Usage

**Memory Per Job:**

- Extraction job: ~5KB (chat messages + AI response)
- Webhook job: ~2KB (payload)

**Retention Policy:**

- Completed jobs: 24 hours
- Failed jobs: Forever (DLQ)

**Estimated Memory:**

- 10,000 completed jobs: ~50MB
- 100 failed jobs: ~500KB

**Redis AOF File Size:**

- Grows unbounded without compaction
- Run `BGREWRITEAOF` periodically to compact

## Troubleshooting

### Job Stuck in "Waiting" State

**Cause:** No active workers or rate limit reached.

**Fix:**

1. Check worker health: `docker-compose ps worker`
2. Check rate limiter: `GET /queue/health` (if `active=0` and `waiting>0`, rate limit is blocking)

---

### Job Fails Immediately (0 retries)

**Cause:** `UnrecoverableError` thrown in job processor.

**Fix:** Check job failure reason in DLQ (`GET /admin/dlq`).

---

### Redis Out of Memory

**Cause:** Too many jobs retained or AOF file too large.

**Fix:**

1. Reduce retention: `removeOnComplete: { age: 3600 }` (1 hour)
2. Clear old jobs: `await extractionQueue.clean(3600 * 24, 1000, "completed")`
3. Compact AOF: `redis-cli BGREWRITEAOF`

---

### Webhook Delivery Fails with Timeout

**Cause:** Webhook endpoint is slow (>10s).

**Fix:** Increase timeout in `queueService.ts:202`:

```typescript
signal: AbortSignal.timeout(30_000), // 30s timeout
```
